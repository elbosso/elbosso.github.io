NullPointer in FontConfig in Alpine-Java-Docker

vorhergehende Artikel in: Java Docker Linux Virtualisierung
30.08.2020

Ich habe nach zwei Tagen herumprobieren doch wieder ein Ticket auf GitHub eröffnet.

Ich habe die Microservices zur Erzeugung von Testdaten wieder einmal überarbeitet - unter anderem wurden neue Generatoren hinzugefügt - wobei mir auffiel, dass zwei der Generatoren nicht korrekt funktionierten, wenn ich die Docker-Container auf dem Raspberry Pi ausführte: Beide Generatoren sorgten für Stacktraces, die für mich zunächst so aussahen, als ob sie auf das Problem zurückzhuführen seien, dass die Anwendung auf einem headless System ausgeführt wurde. Dem widersprach die Tatsache, dass auf einem x86-Docker-Server alles korrekt funktionierte und auch die Tests, die mit -Djava.awt.headless=true ausgeführt wurden keine Probleme erkennen ließen. Als ich daraufhin imImternet recherchierte erkannte ichm, dass ich nicht der Einzige war, der mit solchen Problemen zu kämpfen hatte. Ich habe unten einige der von mir besuchten Suchtreffer angehängt. Allen war gemeinsam, dass es Probleme in Alpine-basierten Docker-Containern gab. Oft wurden Workarounds ins Spiel gebracht, die mir aber alle nicht halfen. Das erklärte aber zumindest, warum die Probleme nicht in den x86-Containern auftauchten - dort hatte ich ein ganz anderes Basis-Image benutzt als für die armhf-Container - dieses beruhte auf Alpine.

Also versuchte ich, den Test mit einem anderen Basis-Image durchzuführen. Ich befürchtete, dass ich dazu wieder stundenlang suchen müsste um ein entsprechendes Image für armhf zu finden - ich war überrascht, dass es vom selben Hersteller eins gab, das auf Debian aufbaute.

Nachdem ich damit getestet hatte und feststellen konnte, dass alles funktionierte habe ich neue Releases für die Microservices erstellt und ein Issue im Projekt eingestellt. Der Vollständigkeit halber folgt hier nocheinmal der vollständige Stacktrace und die Seiten, die mir gezeigt haben, dass ich mit diesem Problem nicht alleine war:
